# 阶段一至二交付报告

## 交付结论

阶段一和阶段二已在本仓库从零实现，没有复制其他项目代码。

当前实现提供以下本地文件协议闭环：

```text
规格 → 计划 → 工作 → 验证 → 迭代 → 验收 → 交付
```

## 系统架构

```text
命令行入口
  → 严格的 Markdown/JSON Schema
  → 合法生命周期转换
  → 预备事务日志与原子重命名
  → TASK_STATE.md 与 STATE_HISTORY.jsonl
  → 连续编号的 Round 产物
  → Evidence 哈希校验
  → Delivery 验收标准映射

阶段二运行层
  → BUDGET.md
  → LOOP_LEDGER.jsonl
  → Guard
  → 自动生成 RUN_LOG.md
  → 自动生成 RUN_SUMMARY.md
```

`TASK_STATE.md` 是当前生命周期的唯一事实源；`STATE_HISTORY.jsonl` 是完整性轨迹，用于拒绝伪造的生命周期修改；`LOOP_LEDGER.jsonl` 是 Attempt 历史事实源，Run Log 和 Summary 都是可确定性重建的投影。

## 命令行能力

| 命令 | 用途 |
|---|---|
| `init` | 创建 Light、Standard 或 Heavy 任务模板 |
| `check` | 校验 Schema 和跨文件不变量 |
| `status` | 读取权威生命周期状态 |
| `next` | 在不修改状态的前提下显示下一步合法操作 |
| `plan` | 校验契约并将状态从 draft 推进到 planned |
| `round` | 创建连续编号的新 Round；迭代后先运行 Guard |
| `verify` | 附加带哈希的 Evidence，并按通过或失败推进状态 |
| `deliver` | 重新计算验收标准映射并推进到 delivered |
| `runtime-init` | 原子初始化预算、账本、日志和摘要 |
| `attempt` | 追加一条严格且连续的 Attempt |
| `guard` | 返回 continue、stop 或 needs_user |
| `summary` | 重新生成纯事实运行摘要 |

## 文件契约

权威 Schema 和运行规则定义在 [DES-001](../03-design/DES-001-file-contract-lifecycle-acceptance.md) 与 [DES-002](../03-design/DES-002-ledger-guard-recovery.md)，并通过严格 Zod 对象执行。未知字段、YAML 重复键、Ledger JSON 重复键以及任务契约、Round 和错误指纹中的占位内容都会被拒绝。

Evidence 绑定以下事实：

- Evidence ID；
- Task ID；
- 当前 Round；
- 代码 revision；
- artifact 路径；
- artifact SHA-256；
- 结果退出码；
- 创建时间。

## 状态机

```text
draft --plan--> planned --round--> working
working --verify pass--> verifying --deliver--> delivered
working/verifying --verify fail--> iterating --round + Guard--> working
```

每次转换都会增加 `state_version`。直接编辑状态会与 CLI 历史产生分歧，并在 `check` 时失败。

## 故障恢复

多文件命令先写入全部临时文件，再持久化 prepared 事务日志，最后通过原子重命名向前完成事务。所有修改和检查操作在读取状态前都会恢复未完成事务。如果事务文件缺失或哈希不一致，系统严格失败，不猜测状态。

## 自动化验证

测试覆盖：

- Guard 决策；
- Standard 和 Heavy 端到端交付；
- Heavy 独立 Verifier 和人工门禁；

## Dogfood 结果

### Standard 任务

[`standard-final`](phase-1-2/dogfood/standard-final/DELIVERY.md) 在 Round 1 到达 `delivered`。Evidence 是最终自动化测试套件产生并经过哈希验证的 artifact；早期 `standard-phase1` 作为开发历史保留。

### Heavy 任务

[`heavy-final`](phase-1-2/dogfood/heavy-final/DELIVERY.md) 在 Round 2 到达 `delivered`；早期 `heavy-phase2` 作为开发历史保留：

1. Round 1 独立拒绝了过期的最终前 Evidence；
2. Guard 在预算内返回 `continue`；
3. Round 2 从最终实现和完整测试生成新 Evidence；
4. 独立验证和明确的人工检查签署当前 Round；
5. Delivery 将所有 Heavy AC 映射到当前 EV-2。

## 已知边界

- 阶段一至二不调用 Agent，也不编排 Prompt；
- 没有后台调度器或多任务队列；
- 没有 Git worktree 管理、push、PR 或 merge；
- 不写外部 Connector；
- 文件系统权限不是安全沙箱；
- 状态历史用于一致性校验，不是抵抗恶意全量重写的密码学签名；
- JSON 重复键检测有意限定在扁平 Ledger Attempt 对象；
- Evidence code revision 由调用方提供，后续 Git 集成负责读取并验证真实 HEAD；
- 人工检查是结构化声明，不是基于身份系统的认证。
