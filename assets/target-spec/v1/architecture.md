# 总体技术架构

> 本文件描述项目级、跨产品与跨特性的技术架构，是所有 Design 文档的共同技术基线。局部设计若扩展或偏离本文件，必须记录原因和影响。

- 状态：草稿
- 负责人：项目维护团队

## 架构目标

初始架构只建立治理边界：业务代码是实现事实源，`spec/` 是业务规格事实源，Spec-Loop 控制目录只保存执行状态。具体系统边界和质量目标由首个获批 Design 补充。

## 架构原则与约束

| 类别 | 原则/约束 | 原因 |
|---|---|---|
| 事实源 | 一个概念只保留一个权威定义 | 防止实现、规格和控制状态分叉 |
| 追踪 | Design 引用 Feature 和本总体架构，Task 引用 Design | 保持变更影响可追踪 |
| 安全 | 初始化只补缺，不覆盖已有规格 | 保护项目已有决策与记录 |

## 系统上下文

项目的用户、外部系统和主要交互应在相关 Product、Feature 与 Design 中定义，并在形成跨特性约束时回写本文件。

## 核心组件与数据流

首个涉及系统结构的获批 Design 负责登记组件职责、依赖方向和关键数据流；登记前不得把未经批准的示例架构视为事实。

## 数据、接口与部署

数据所有权、接口协议、部署拓扑、配置与密钥边界由相关 Design 定义。形成项目级共同约束后，必须同步到本文件。

## 横切质量要求

性能、可用性、安全与隐私、可观测性、可维护性和成本目标必须可验证；具体阈值和验证方式由对应 Feature 与 Design 批准。
